# HARNESS.md

이 파일은 `AGENTS.md` §43(모델 하네스)·§44(목표 완결과 완성 기준)의 전체 절차 원문입니다. §43.9 goal 프로토콜이 적용되는 코드·기능 변경 작업(사소한 한 줄 수정, 순수 질의응답, 즉시 자명한 기계적 편집은 예외)에서는 §4 작업 시작 전 필수 절차에 따라 이 파일 전문을 읽고 그대로 적용합니다.

이 하네스(§43)는 사용자의 에이전틱 코딩 툴 전역 설정 5곳(`~/.claude/CLAUDE.md`, `~/.codex/AGENTS.md`, `~/.config/opencode/AGENTS.md`, `~/.config/kilo/AGENTS.md`, `~/.zcode/AGENTS.md`)에도 동일한 내용으로 동기화되어 있습니다. 이 저장소가 하네스 원본이며, 한 곳을 고치면 나머지도 함께 갱신해야 합니다. 전역 설정이 없는 협업자나 다른 환경에서는 이 파일이 하네스를 적용받는 유일한 경로입니다. §44는 아직 전역 5곳에 동기화되어 있지 않습니다(`BACKLOG.md` 참고).

## 43. 모델 하네스 (Model-Tier Harness)

이 저장소의 작업 품질 기준은 **실행 모델의 등급과 무관하게 동일**합니다. 더 강한 모델에서 더 가볍고 판단력이 약한 모델로 전환되더라도, 그 차이는 아래의 절차적 엄격함으로 보정합니다. 이 섹션은 강한 모델이 암묵적으로 수행하는 작업 습관을 **모든 모델의 명시적 의무**로 고정한 것입니다.

이 하네스는 §1–§42를 대체하지 않습니다. 각 절은 관련 원 조항을 가리키며, 하네스가 하는 일은 그 규칙을 **모델 등급과 무관하게 항상, 생략 없이 적용**하도록 강제하는 것입니다. 규칙 본문의 단일 진실 공급원은 언제나 원 조항이며, 문구가 서로 어긋나면 원 조항이 우선합니다. 특정 제품·모델 이름은 등급이 재편되면 낡으므로, 이 섹션은 절대적 모델명 대신 상대적 등급('가장 강한 모델', '경량 모델')으로 기술합니다.

### 43.1 계획 분해 (Plan Decomposition)

- 요청을 받으면 코드를 수정하기 전에, 검증 가능한 작은 단계(micro-task)로 분해한 계획을 먼저 작성합니다.
- 각 단계는 하나의 목적만 가지며, 단계별 완료 기준을 함께 정의합니다.
- 3단계 이상이 되는 작업은 체크리스트(todo)로 관리하고, 단계가 끝날 때마다 상태를 갱신합니다. todo 도구가 없는 환경에서는 작업 노트 파일에 체크리스트를 기록합니다.

### 43.2 근거 우선 (Evidence over Memory)

§3의 근거·반론 원칙을 코드 작업에 그대로 적용합니다. 하위 모델에서도 다음을 생략하지 않습니다.

- 파일 내용·함수 시그니처·경로·설정 키·CLI 옵션을 기억이나 추측으로 단정하지 않고, 수정 대상과 그 주변(호출부·참조부·관련 테스트)을 실제로 읽은 뒤에만 수정합니다.
- 존재를 확인하지 않은 API·파일·옵션·의존성을 사용하지 않습니다. 확신이 없으면 사용자에게 묻기 전에 명령 실행·코드 검색·문서 확인으로 스스로 검증합니다.
- 확인한 사실이 사용자의 주장과 다르면 동의하는 대신 근거와 함께 반대 의견 또는 대안을 제시합니다(§3).

### 43.3 작은 변경과 즉시 검증 (Small Steps, Fast Verification)

- 한 번에 크게 바꾸지 않습니다. 한 단계를 바꾼 뒤 가능한 가장 빠른 검증(린트·타입 체크·관련 테스트, §14)을 실행해 통과를 확인하고 다음 단계로 갑니다. 검증 없이 여러 단계를 연속으로 진행하지 않습니다.
- 상호 의존적인 변경(시그니처 변경과 그 호출부 수정 등)은 하나의 단계로 묶어 함께 검증합니다.
- 실행할 검증 명령이 없는 작업(문서 등)은 결과물을 처음부터 다시 읽고 요구사항과 1:1로 대조하는 것으로 검증을 대신하며, 무엇으로 검증했는지 보고에 명시합니다.

### 43.4 상태 외부화 (Externalized Working State)

- 긴 작업은 진행 상황·확인된 사실·남은 작업을 `History Document`, `Task Document` 또는 작업 노트(예: `PROGRESS.md`)에 기록하며 진행합니다(§17).
- 세션이 중단되거나 컨텍스트가 잘려도 기록만 보고 이어서 작업할 수 있는 상태를 유지합니다.

### 43.5 자기 검증 루프 (Mandatory Self-Review)

- 완료를 선언하기 전에 전체 diff를 처음부터 다시 읽고 §16 Pre-Commit Review 체크리스트를 수행합니다.
- 요구사항과 구현 결과를 1:1로 대조하고, 요구사항에 대응하지 않는 변경 줄은 되돌립니다.
- 하위 모델도 이 단계를 생략하지 않습니다. 자기 검증의 실제 근거(명령 출력 또는 대조 결과) 없이 "완료"를 선언하지 않습니다.

### 43.6 실패 처리 (No Silent Give-Up)

- 에러가 나면 로그 전체를 읽고 원인을 좁혀 수정한 뒤 재시도합니다. 원인 분석 없이 같은 명령을 그대로 재실행하지 않습니다(§39).
- 검증 통과 자체를 목적으로 테스트를 약화·삭제·skip하거나 기댓값을 실제 출력에 맞춰 고치지 않습니다. 테스트 자체의 결함이 원인일 때만 그 근거를 커밋 메시지 또는 `History Document`에 기록한 뒤 수정합니다(§14).
- 같은 접근이 3회 실패하면 접근 방식을 바꾸고, 그래도 안 되면 시도 내역·실패 로그·남은 가설과 함께 `BLOCKED`로 보고합니다(§26·§40). "아마 해결되었을 것" 상태로 작업을 종료하지 않습니다.

### 43.7 범위 준수 (Scope Discipline)

- 모든 변경 줄은 요청된 작업, 검증 실패 수정, 또는 이번 변경으로 직접 발생한 정리에만 연결되어야 합니다(§7). 요청되지 않은 리팩터링·전체 포맷팅·기능 추가는 하지 않습니다.
- 발견한 개선점은 고치지 않고 후속 작업으로 기록합니다.

### 43.8 모델 사용 전략 (Model Allocation)

- 가능하면 계획 수립·설계 판단·최종 리뷰는 사용 가능한 **가장 강한 모델**로, 기계적 구현·반복 수정·문서 갱신은 **경량 모델**로 수행합니다.
- 이 배분은 세션을 구성하는 사용자, 또는 하위 에이전트에 모델을 지정할 수 있는 오케스트레이터를 위한 권고입니다. 자신의 모델을 선택할 수 없는 실행 에이전트에게 모델 변경 의무를 부과하지 않습니다.
- 경량 모델로 실행되는 에이전트도 §43.1–§43.7을 하나도 생략하지 않습니다. 실행 모델의 등급을 낮은 품질·생략된 검증·축소된 범위의 사유로 삼지 않습니다.

### 43.9 goal 실행 프로토콜 (Goal Execution Protocol)

코드·기능을 바꾸는 "goal" 작업은 **분석 → 구현 → 감사**의 세 역할을 거칩니다(사소한 한 줄 수정, 순수 질의응답, 즉시 자명한 기계적 편집은 예외). 이 프로토콜은 §43.1–§43.8을 대체하지 않고 그 위에서 실행 구조를 정합니다.

**단일 모델 환경(기본형)**에서는 하나의 세션이 세 역할을 순차적으로 갈아입으며 수행하고, 각 역할의 산출물(1단계 계약, 3단계 감사 근거)을 명시적으로 남깁니다. 이 기본형은 모든 환경에 항상 적용됩니다. **하위 에이전트에 모델을 지정할 수 있는 오케스트레이터가 있으면(최적화형)**, 분석·감사는 가장 강한 모델에, 구현은 경량 모델(서로 독립적인 영역은 병렬)에 위임해 같은 세 역할을 나눠 맡깁니다. 어느 쪽이든 세 역할과 그 산출물은 동일합니다.

- **1단계 · 분석·설계 (강한 역할)**: 수정 전 코드베이스를 실제로 읽어 데이터 흐름·호출부·저장 구조를 파악하고, "바로 수정 가능한 수준"의 구현 지점(파일·함수·앵커·데이터 계약)을 특정합니다(§43.2). 이름과 실제 데이터가 어긋나는 등 확인이 필요한 지점은 근거와 함께 먼저 사용자에게 확인합니다(§3). 결과를 계약(변경 파일·함수 시그니처·데이터 형태·시드 값)으로 고정해 이후 단계의 기준으로 삼습니다.
- **2단계 · 구현 (경량 역할)**: 1단계에서 고정한 계약과 정확한 앵커·코드에 따라 변경을 적용합니다. 서로 독립적인 파일·영역은 병렬로 처리할 수 있으며, 각 작업 단위는 자기 영역만 편집하고 가능한 즉시 검증합니다(§43.3). 경량 모델이라도 §43.1–§43.7을 생략하지 않습니다.
- **3단계 · 감사 (강한 역할)**: 구현 결과를 신뢰하지 않고 전체 diff를 다시 읽어 1단계 계약·요구사항과 1:1로 대조하고(§43.5·§16), 실행 가능한 검증(구문 체크·핵심 로직 실측·런타임/브라우저 확인 등)으로 실제 동작을 확인한 뒤에만 완료를 보고합니다(§27·§40). 계약과 어긋나거나 범위를 벗어난 변경(§43.7)은 되돌리거나 바로잡습니다.
- 최종 보고에는 세 역할을 어떤 구성으로 수행했는지(단일 모델 순차 분리인지 다중 모델 위임인지)와 3단계 감사의 실제 검증 근거를 함께 명시합니다(§27).

## 44. 목표 완결과 완성 기준 (Goal Completion & Doneness)

이 섹션은 자동 완주(§2)의 반대편 함정 — 모든 아이디어를 활성 작업으로 만들고, 완료된 작업을 무한히 다시 여는 것 — 을 막는 규칙입니다. §5·§7의 범위 규칙과 §40의 완료 기준 위에서 적용합니다.

### 44.1 목표 완결 우선 (Primary Objective)

- 프로젝트 확장보다 현재 목표의 완결이 먼저입니다. 크고 사변적인 설계보다 작고 완결되고 검증 가능한 결과를 우선합니다.
- 유용해 보이는 아이디어를 전부 활성 기능·프로젝트로 만들지 않습니다. `Backlog Document`에 기록하는 것으로 충분합니다.
- `Goals Document`가 있으면 활성 목표의 작업만 수행합니다. 관련 코드를 수정 중이라는 이유로 미래 목표를 앞당겨 구현하지 않으며, 우선순위 변경은 사용자가 결정합니다.

### 44.2 Production-Ready 기본값

- 일회성 프로토타입을 명시적으로 요청받지 않는 한, 프로덕션 사용을 전제로 설계·구현합니다: 유지보수 가능한 구조, 명확한 모듈 경계, 신뢰할 수 있는 오류 처리, 보안·개인정보 통제, 데이터 무결성과 복구, 테스트 가능성, 필요한 경우 관측 가능성, 업그레이드·마이그레이션 경로, 운영상 제약의 문서화(placeholder·가짜 성공 금지는 §6).
- Production-ready는 **모든 미래 기능의 선구현이 아닙니다.** 프로덕션으로 가는 경로를 보존하는 점진적 목표를 사용합니다.

### 44.3 신규 발견 작업 분류 (Work Classification)

- 작업 시작 전에 요청된 결과물, 수용 기준(acceptance criteria), 명시적 제외 범위, 검증 방법, 완료를 막는 조건을 식별합니다(§5).
- 작업 중 발견한 새로운 작업은 **Blocker**(없이는 완료가 불가능·불안전) / **Current scope**(수용 기준이 직접 요구) / **Backlog**(유용하나 지금은 불필요) / **Rejected**(불필요·중복·유해) 중 하나로 분류합니다.
- Blocker와 Current scope만 진행 중인 작업을 확장할 수 있으며, Blocker는 확장 전에 사용자에게 설명합니다. Backlog는 `Backlog Document`에 기록만 하고, 명시적인 우선순위 지정 전에는 구현하지 않습니다.
- 개선을 시도하는 과정에서 작업 범위를 조용히 넓히지 않습니다(§7).

### 44.4 의사결정 정책 (Decision Policy)

- 되돌리기 쉬운 결정은 가장 단순하고 합리적인 선택지와 기존 관례로 빠르게 구현·검증하고, 장시간 비교하지 않습니다.
- 되돌리기 어려운 결정(보안, 데이터 형식, 공개 API, 인프라, 라이선스, 주요 의존성)은 현실적인 대안을 비교하고, 트레이드오프를 명시하고, `Decision Log`에 기록하고, 가능하면 마이그레이션·롤백 전략을 정의합니다.
- 새 증거가 전제를 실질적으로 바꾸지 않는 한 수용된 결정을 다시 열지 않으며, 더 새롭거나 흥미롭다는 이유만으로 동작하는 기술을 교체하지 않습니다.

### 44.5 아키텍처 최소주의 (Minimal Architecture)

- 확인된 요구사항을 안전하게 지탱하는 가장 작은 아키텍처를 사용합니다(사변적 추상화 금지는 §6).
- 확장 포인트는 현재 요구사항이 복수의 구현을 필요로 하거나, 알려진 단기 목표가 그 변형을 요구하거나, 나중에 바꾸면 상당한 마이그레이션 위험이 생길 때만 만듭니다.
- enterprise답게 보이게 하려는 목적만으로 인터페이스·계층·서비스·모듈을 만들지 않습니다. 데이터·도메인 로직·인프라·UI의 경계는 프로젝트 규모가 정당화하는 범위에서 유지합니다.

### 44.6 리뷰 상한 (Review Limits)

- 의도적인 리뷰는 최대 2회가 기본입니다 — 1차(§16 연계): 정확성, 수용 기준, 보안·데이터 안전, 실패 처리 / 2차(§33 연계): 유지보수성, 회귀 위험, 문서, 최종 검증.
- 예외는 치명적 결함 발견, 수용 기준 변경, 이전 결정을 무효화하는 새 증거뿐입니다.
- 2차 이후 발견되는 non-blocking 개선점은 완료된 작업을 다시 열지 말고 `Backlog Document`로 보냅니다. 안심을 얻으려는 목적만으로 광범위한 리뷰를 반복하지 않습니다.

### 44.7 완료의 유효성 (Completion Validity)

- §40 충족, 치명적 결함 없음, 남은 non-blocking 작업의 별도 기록, 빌드·복구 가능한 프로젝트 상태면 완료입니다.
- **완벽함은 완료 기준이 아니며, 발견된 개선점이 완료를 자동으로 무효화하지 않습니다.** 완료 후 발견분은 `Backlog Document`로 보냅니다(§44.3·§44.6).

### 44.8 제품 가치 검증 (Product Validation)

- 기술적 가능성이 제품 가치를 증명하지 않습니다. 제품 결정에서는 기술, 사용자 수요, 유통, 운영, 법·정책 위험을 구분합니다.
- 실질적인 새 기능을 만들기 전에 사용자 문제, 문제가 실재한다는 근거, 기대하는 사용자 행동, 성공 측정 방법, 가장 저렴한 검증 방법을 식별합니다.
- 불확실한 가정을 실제 사용자로 검증하는 일을 회피하기 위해 엔지니어링을 더 쌓지 않습니다.

### 44.9 운영 현실 (Operational Reality)

- 인프라 비용이 현재 0이라는 이유로 서비스를 "사실상 무료"라고 설명하지 않습니다. 지원 부담, 남용 방지, 모니터링, 백업, 장애 대응, 플랫폼 정책 변화, 데이터 보존, 법적 책임, 유지보수 시간을 함께 고려합니다.
- 신뢰성·규모·모더레이션 요구가 무비용 제약과 충돌하면 트레이드오프를 직접 설명합니다.

### 44.10 확장 중지 조건 (Stop Conditions)

§26 중단 조건이 위험 상황에서 멈추는 규칙이라면, 이 조건은 잘 되어가는 작업의 확장을 멈추는 규칙입니다. 수용 기준 충족, 남은 발견 사항이 전부 non-blocking, 추가 작업이 확인된 요구가 아닌 가상의 요구 대상, 리뷰를 반복해도 릴리즈 결정이 바뀌지 않음, 새 아이디어가 미래 목표 소속 — 이 중 하나라도 해당하면 현재 작업을 마무리하고 추가 아이디어는 별도로 기록합니다.
